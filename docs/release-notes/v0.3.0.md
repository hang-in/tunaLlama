# v0.3.0 - production-RAG path 도입 (2026-05-11)

검색 품질이 측정 자산 위에서 의미 있게 올라왔다. **HyDE hybrid 가

## 환경변수

- `TUNA_EMBEDDING_MODEL`: HF model id. default `BAAI/bge-m3`. Korean-heavy
  corpus 면 `nlpai-lab/KURE-v1` 권장 (drop-in, dim 1024 호환).
- `TUNA_EMBEDDING_DEVICE`: cpu / mps / cuda. macOS 일상 사용엔 cpu 권장.
- `OLLAMA_CLOUD_API_KEY`: cloud-augmented path (expanded / normalized / HyDE)
  사용 시.

## delegation 메시지 정정

초기 가설 "토큰 절약" 은 측정 보류 (Anthropic API 미보유). Phase 4-4 + 5-3
측정에서 isolated task 에선 cloud LLM 위임과 native 의 코드 품질 거의
동일 (kw_hit 0%) - delegation 의 가치는:

1. **메인 conversation 컨텍스트 격리** (긴 코드/리뷰가 메인 대화에 안 들어옴)
2. **무거운 작업의 cloud 분리** (대형 분석, large refactor, batch)
3. **로컬 모델 활용** (개인 정보 / 외부 송신 회피)

## abandon

- Phase 5-2D **MMR** - 우리 use case (relevant 가 paraphrase set) 에서
  anti-pattern. 외부 권고는 긴 document + 무관 후보 다수 환경 가정.
- Phase 5-4 **token measurement** - Anthropic API 미보유. `-p` mode
  transcript 파싱 가성비 부족.

## dogfooding 핵심 통찰

- **9 회 (Round 7-15) standalone-toy 실패** 는 spec 형식이 아닌 **위임 채널**
  문제. `tuna_dev_review_from_spec` 의 generate→review 흐름이 코드 모드 강제.
- **Round 16** 에서 `tuna_general_task` (catch-all) 채널로 swap → 첫 차용
  가능 결과. bounded data tasks (시드 / 알고리즘 초안 / prompt variants) 에
  적합.

## 한계 (정직)

- HyDE 0.92 P@1 은 **시드 record 가 task description 형식**이라 hypothetical
  answer 와 매칭 매우 강함. 실 사용 환경 (record 가 다른 형식) 에서 효과
  편차 가능.
- σR@5 0.16 = 외부 목표 0.15 거의 도달, 단 R@5 0.50 은 RAG 표준 0.8 미달.
  사람 in-the-loop UX (5 후보 surface) 에는 충분.
- `auto_recall=always` 는 default 가 아니다 - cloud LLM 이 무관 prefix
  무시한다는 측정 결과 있지만 코드 품질 저하 risk 가 0 인 건 아님.
