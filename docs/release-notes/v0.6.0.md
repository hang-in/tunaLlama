## Phase 9 - 임베딩을 Ollama 로 (torch-free 코어)

### 동기
- torch(+cpu) 는 GPU 미사용 + **Python 3.13 Windows 에서 `import torch` 무한 hang**.
- 설치 용량이 크고 CPU/GPU 빌드 관리가 번거로움.
- Secall 이 검증: `qwen3-embedding:0.6b` (Ollama, 1024-dim) 이 bge-m3 의 NaN 버그 해결.

### 변경
- `tunallama_core/memory/vector.py` `embed()` 재작성: `ollama.Client.embed()` 호출 후
  L2 정규화. dim≠1024 / 호출 실패는 `LLMError`. 시그니처·`EMBEDDING_DIM`·blob 포맷 불변.
- 기본 임베딩 모델 **`qwen3-embedding:0.6b`** (Ollama 태그). host 는 `[llm.ollama].host`
  재사용. `MemoryConfig.embedding_model` 필드 추가.
- `pyproject`: `sentence-transformers` 를 코어 deps 에서 **`[rerank]` optional extra** 로.
  → 기본 `pip install -e .` 는 **torch-free** (검증: import 체인에 torch 전무).
  cross-encoder reranker 는 `[rerank]` 설치 시에만 활성 (`search.py` graceful degrade).
- GPU: Ollama 가 자동 관리 (VRAM 충분 시 GPU 임베딩).

### 마이그레이션
- 기존 `memory.db` 의 BGE 벡터는 Qwen 공간과 비호환 → 재임베딩 필요(신규는 자동).

## Windows 지원 - MCP 를 HTTP 전송으로

### 문제
Windows 에서 **Python + stdio MCP 서버가 Claude Code 와의 조합에서 in-session tool
호출이 무한 hang** 한다. 서버 자체는 정상(외부 클라이언트로는 1~2초 응답, 같은 머신서
HTTP/비-Python MCP 는 정상 작동) — Claude Code 의 Windows stdio MCP 클라이언트 이슈.
실측: `tuna_load_memory` 가 stdio 로 **565초 무응답**.

### 해결 - streamable-http 데몬
- `mcp_server.main()`: `TUNA_MCP_TRANSPORT=http` (또는 sse) 로 전송 전환 (기본 stdio).
  `TUNA_MCP_HOST`/`TUNA_MCP_PORT`(기본 127.0.0.1:8766). FastMCP 내장.
- `plugin/bin/tunallama-httpd.cmd`: HTTP 데몬 런처.
- `plugin/bin/tunallama-win-setup.ps1`: **한 번 실행** 셋업 (idempotent) —
  로그온 자동시작 데몬 + `claude mcp add --transport http tunallama` + 플러그인 stdio
  MCP 비활성화. `-Uninstall` 지원.
- **mac/Linux 는 stdio 그대로** (자동 spawn, 무변경).
- 검증: `mcp__tunallama__tuna_load_memory` / `tuna_recall` in-session 정상 (즉시 응답).
